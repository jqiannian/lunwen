## 1. 原始用户需求
- 来源：`docs/references/user-stories/最简模型2.md`、`docs/references/user-stories/论文提纲.md`
- 背景：需要在流程性交通场景中进行无监督异常检测，首要示例为“红灯停”规则。
- 用户痛点：
  - 缺少高质量的数据处理管道及可扩展的规则注入框架。
  - 现有模型对交通规则的理解不足，难以解释违规原因。
  - 需要减少人工标注，利用自训练与记忆模块巩固正常模式。
- 业务目标：
  - 构建可复用的交通违规检测引擎，以最小 MVP “红灯停”检测为起点，逐步扩展到多规则、多场景。
  - 融合真实数据集（BDD100K、Cityscapes）与合成数据，形成统一数据加载与验证流程。
  - 提供自训练、关系推理、语义注入等机制以提升准确率及可解释性。
- 关键约束：
  - 严格遵循交通法规语义（如红灯停、车速限制等）。
  - 模型训练需可在有限资源下完成（单机 NVIDIA RTX 4090，CUDA 12.1），默认 Python 3.11 + PyTorch 2.4。
  - 核心模块（数据、模型、规则、注意力解释）全部以 Python/Poetry 交付。
  - 所有测试需具备可复现的数据准备脚本。

### 1.1 重点方向与技术栈约束
- **A. 交通规则合规（红灯停优先）**：MVP 必须覆盖红灯停场景，输出违规证据链（停止线距离、速度、灯色）。
- **B. 语义注入 / 关系推理**：利用图神经网络 + 规则 DSL，对车辆、交通灯、停止线等实体建模，维持语义一致性。
- **C. 注意力增强与可解释性**：要求在关系推理层中显式采用注意力机制（如多头 GAT、记忆检索注意力），并输出注意力热力图/权重供业务诊断。
- **技术栈**：Python 3.11、Poetry、PyTorch 2.4（CUDA 12.1 变体）、torchvision、OpenCV、pydantic、prometheus-client；禁止混用未审查语言或脚本。GPU 版 PyTorch 需通过 `pip install torch==2.4.1+cu121 torchvision==0.19.1+cu121 torchaudio==2.4.1+cu121 --index-url https://download.pytorch.org/whl/cu121` 安装。

### 3.2 非功能需求
- 性能：单次训练轮在 4×A100 GPU 下 <= 2 小时；推理每帧延迟 <= 120ms（离线批处理场景）。
- 准确性：红灯违规检测召回率 ≥ 0.9（MVP synthetic 数据）；后续真实数据 AUC ≥ 0.85。
- 可扩展性：规则引擎需支持 ≥5 条规则；数据加载器可水平扩展到 1M 样本。
- 可维护性：配置项集中在 `.env`/`pyproject.toml`；日志、指标具备统一命名。
- 可复现性：提供 `scripts/setup_mvp_env.sh`（待实现）用于环境还原。

## 4. 影响分析
- 系统/模块：数据摄取、训练调度、规则管理、可解释性服务均为新增模块。
- 第三方依赖：PyTorch、torchvision、OpenCV、BDD100K 官方解析库、Prometheus client。
- 风险与缓解：
  - 数据许可风险 → 仅使用公开数据并记录来源。
  - 规则冲突 → 在规则 DSL 中加入冲突检测、优先级字段。
  - 计算资源不足 → MVP 先运行 synthetic 数据，后续引入分布式训练计划。

## 6. 验收标准与测试入口
- MVP（ITER-2025-01）：执行 `poetry run python tools/train_red_light.py --config configs/mvp.yaml`，并运行 `poetry run python tools/test_red_light.py --scenario all`。输出需包含 3 个示例场景、违规判定截图、注意力热力图与停止线距离/速度记录。
- 集成测试：`pytest tests/integration -m traffic_rules` 覆盖所有业务约束。
- 验收数据集：合成 100 个场景 + BDD100K 10 个样本（后续迭代补充）。
- 环境还原脚本：`scripts/setup_mvp_env.sh`、`scripts/prepare_synthetic_data.py`、`scripts/render_attention_maps.py`（待实现，需在测试文档中登记）。
